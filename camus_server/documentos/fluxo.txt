FLUXO DE AUTENTICACAO — CAMUS SERVER


1. CADASTRO  (POST /cadastrar)


Cliente envia:
{
  "nome":  "exemplo",
  "email": "exemplo@email.com",
  "senha": "minhasenha123"
}

Servidor:

  1. Valida que nome, email e senha estao presentes.
  2. Verifica se o e-mail ja existe no banco (campo UNIQUE).
  3. Gera hash da senha usando bcrypt (custo 12).
  4. Salva { nome, email, senha_hash } na tabela users.
  5. Gera um codigo de verificacao de 6 digitos.
  6. Salva o codigo na tabela auth_codes com validade de 10 minutos.
  7. Envia o codigo para o e-mail cadastrado.
  8. Retorna:

HTTP 201

{
  "requires_2fa": true,
  "challenge_id": 1,
  "message": "Codigo enviado para o email"
}


Erros possiveis:

400 — campo obrigatorio ausente  
409 — e-mail ja cadastrado  
429 — muitas requisicoes (rate limit)  
500 — falha interna



2. LOGIN  (POST /login)

Cliente envia:

{
  "email": "exemplo@email.com",
  "senha": "minhasenha123"
}

Servidor:

  1. Busca o usuario pelo e-mail na tabela users.
  2. Compara a senha enviada com o hash armazenado.
  3. Se invalido, retorna 401.
  4. Gera um codigo de verificacao de 6 digitos.
  5. Salva o codigo na tabela auth_codes com validade de 10 minutos.
  6. Envia o codigo para o e-mail cadastrado.
  7. Retorna:

HTTP 200

{
  "requires_2fa": true,
  "challenge_id": 5,
  "message": "Codigo enviado para o email"
}


Erros possiveis:

400 — campo obrigatorio ausente  
401 — credenciais invalidas  
429 — muitas requisicoes (rate limit)  
500 — falha interna



3. VERIFICACAO DO CODIGO 2FA  (POST /verificar-codigo)

Cliente envia:

{
  "challenge_id": 5,
  "codigo": "123456"
}

Servidor:

  1. Busca o codigo na tabela auth_codes.
  2. Verifica se:
       - o codigo corresponde
       - nao foi utilizado
       - nao expirou
  3. Marca o codigo como usado.
  4. Busca o usuario associado.
  5. Gera um token JWT de autenticacao.
  6. Cria uma sessao na tabela sessions.
  7. Retorna:

HTTP 200

{
  "ok": true,
  "message": "Autenticacao concluida",
  "token": "<jwt_token>",
  "expires_at": "2026-03-23T13:25:54"
}



4. CONSULTA DO USUARIO AUTENTICADO  (GET /me)

Cliente envia:

Header:
Authorization: Bearer <token>

Servidor:

  1. Valida assinatura do token JWT.
  2. Verifica se a sessao existe na tabela sessions.
  3. Verifica se a sessao nao foi invalidada.
  4. Verifica se a sessao nao expirou.
  5. Retorna os dados do usuario.

HTTP 200

{
  "id": 1,
  "nome": "exemplo",
  "email": "exemplo@email.com"
}



5. LOGOUT  (POST /logout)

Cliente envia:

Header:
Authorization: Bearer <token>

Servidor:

  1. Valida o token JWT.
  2. Busca a sessao no banco pelo token_jti.
  3. Marca a sessao como invalidada.
  4. Retorna:

HTTP 200

{
  "ok": true,
  "message": "Sessao encerrada com sucesso"
}



6. RESUMO DOS ENDPOINTS DE AUTENTICACAO

POST /cadastrar
POST /login
POST /verificar-codigo
POST /logout
GET  /me
