JUSTIFICATIVAS TECNICAS — SEGURANCA DE AUTENTICACAO



1. CRIPTOGRAFIA DE SENHAS — bcrypt

As senhas dos usuarios nunca são armazenadas em texto puro.
Antes de salvar no banco, a senha é processada com bcrypt,
um algoritmo projetado especificamente para hashing de senhas.

O bcrypt foi escolhido porque possui suporte nativo a salt
criptografico e um fator de custo configuravel, tornando ataques
de força bruta significativamente mais caros computacionalmente.

Mesmo que o banco de dados seja comprometido, as senhas
permanecem protegidas na pratica.



Armazenamento do hash + salt

O bcrypt retorna uma unica string de aproximadamente 60
caracteres que ja contem:

- algoritmo
- fator de custo
- salt
- hash da senha

Exemplo de formato:

$2b$12$<salt><hash>

Isso significa que a coluna "senha" da tabela users guarda
tudo que e necessario para verificar a senha no futuro.
Nao e necessario armazenar o salt separadamente.



2. FLUXO DE AUTENTICACAO DOCUMENTADO

O fluxo completo de autenticação com cadastro, login com
autenticação de dois fatores (2FA), criação de sessão e
logout está documentado no arquivo fluxo.txt.

Resumo dos endpoints de autenticação:

POST /cadastrar
POST /login
POST /verificar-codigo
POST /logout
GET  /me



3. AUTENTICACAO DE DOIS FATORES COM E-MAIL (2FA)

Mesmo com uma senha forte, ela pode ser vazada por phishing,
reutilização de senhas ou engenharia social. O segundo fator
garante que conhecer a senha sozinha nao e suficiente para
acessar a conta.

Fluxo utilizado no sistema:

1. Usuario envia email + senha no POST /login.
2. Servidor valida as credenciais.
3. Se corretas, gera um codigo aleatorio de 6 digitos.
4. O codigo e salvo na tabela auth_codes com:
     - id do usuario
     - tipo da operacao (login ou register)
     - tempo de expiracao (10 minutos)
     - status de uso
5. O codigo e enviado para o e-mail cadastrado.
6. O cliente envia o codigo via POST /verificar-codigo.
7. O servidor valida o codigo e gera o token JWT.

Boas praticas adotadas:

- Codigo invalido apos o primeiro uso.
- Expiracao curta (10 minutos).
- Novo codigo invalida o anterior.
- O sistema nao revela se o e-mail existe no sistema
  durante tentativas de login.



4. SESSOES COM TEMPO DE EXPIRACAO

Apos a verificacao do 2FA, o servidor emite um token JWT
com tempo de expiracao configurado.

Além disso, uma sessao e registrada na tabela sessions
contendo:

- user_id
- token_jti
- expira_em
- invalidada

Em cada requisicao autenticada o servidor valida:

- assinatura do token JWT
- existencia da sessao
- se a sessao nao foi invalidada
- se a sessao nao expirou



5. INVALIDACAO DE SESSAO NO LOGOUT

O endpoint POST /logout invalida a sessao atual no banco
de dados. A sessao correspondente ao token utilizado e
marcada como invalidada.

Mesmo que o token ainda esteja dentro do prazo de validade,
ele nao sera mais aceito pelo servidor apos o logout.



6. EVIDENCIAS FUNCIONAIS

As evidencias de funcionamento do sistema de autenticacao
serao coletadas das seguintes formas:

a) Logs do servidor  
Cada requisicao aos endpoints sera registrada com timestamp,
IP de origem e codigo de resposta HTTP.

b) Prints de testes manuais  
Capturas de tela do terminal demonstrando cadastro, login,
verificacao de codigo e acesso autenticado.

c) Inspecao direta do banco  
Consultas SQL demonstrando que as senhas nao sao armazenadas
em texto puro.

d) Teste de rate limit  
Script enviando varias requisicoes consecutivas ao endpoint
/login para demonstrar o retorno HTTP 429.



7. PROTECAO CONTRA FORCA BRUTA — RATE LIMITING

Sem limite de requisicoes, um atacante poderia automatizar
tentativas de login em grande escala.

Para mitigar esse risco, foi implementado rate limiting
utilizando Flask-Limiter.

Limites definidos:

POST /cadastrar        — 5 requisicoes por minuto  
POST /login            — 5 requisicoes por minuto  
POST /verificar-codigo — 10 requisicoes por minuto  

Quando o limite e excedido, o servidor retorna:

HTTP 429 — Too Many Requests

Essa estrategia reduz significativamente ataques automatizados
de tentativa de senha.



8. VISAO GERAL DA DEFESA EM CAMADAS

Camada 1 — Prevencao de acesso em massa  
Rate limiting reduz ataques automatizados antes mesmo
da verificacao de credenciais.

Camada 2 — Protecao da senha em repouso  
bcrypt protege as senhas mesmo em caso de vazamento
do banco de dados.

Camada 3 — Verificacao de identidade com segundo fator  
2FA por e-mail impede que credenciais corretas sejam
suficientes sem acesso ao e-mail do usuario.

Essa combinacao segue o principio de defesa em profundidade,
onde cada camada protege o sistema caso outra falhe.
